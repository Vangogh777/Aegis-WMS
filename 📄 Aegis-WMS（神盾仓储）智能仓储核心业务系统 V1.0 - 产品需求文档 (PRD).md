

**文档版本**：V1.0 (核心重构版)  
**产品定位**：聚焦粮库核心“仓储管理”与“粮情安全”，剥离外围物流流程，打造具备**高并发处理能力**与**高扩展性（IoT Ready）**的现代化数字粮库底座。

---

## 一、 系统整体架构概览
### 1.1 技术栈选型
+ **前端**：Vue3 (PC Web) + Echarts (数据可视化) + WebSocket (实时报警推送)
+ **后端**：Spring Boot / Spring Cloud (Java) + MyBatis-Plus
+ **核心中间件**：
    - **MySQL**：核心业务数据持久化。
    - **Redis**：分布式细粒度锁（保障库存高并发强一致性）、字典缓存。
    - **Kafka**：消息总线，实现核心链路的异步解耦与削峰。

### 1.2 DDD 业务领域划分 (5大核心领域)
1. **基础主数据领域**：确立“库区-仓房-货位”三级空间架构。
2. **出入库库存领域**：极简出入库，聚焦高并发下的精准库存增减。
3. **粮情检测领域**：手工粮情全量记录，Kafka异步触发安全预警。
4. **通风作业领域**：标准化通风状态机，Kafka异步处理能耗测算。
5. **数据与报表领域**：动态库存台账与多维粮情趋势视图。

## 二、 👑 核心底层机制设计（研发重点关注）
### 2.1 多维度库存高并发扣减机制（防超卖/防覆盖）
+ **业务场景**：秋粮收购高峰，多辆运粮车同时在同一个仓房的不同位置（货位）进行卸粮入库，或并发提货出库。
+ **技术方案**：借鉴 `ConcurrentHashMap` 分段锁思想，结合 Redis 与 Spring 事务。
    - **锁的粒度**：下钻至“货位”级别。
    - **Lock Key 定义**：`wh:inv:lock:{warehouseId}_{binId}_{positionId}`
    - **执行生命周期（强制规范）**：
        1. 线程尝试获取 Redis 货位分布式锁（失败则自旋或 Fast-Fail 提示用户稍后重试）。
        2. `获取锁成功` ➔ 开启 Spring `@Transactional` 事务。
        3. DB查询当前货位库存 ➔ 内存校验是否超扣 ➔ 执行 DB `UPDATE` 扣减/增加 ➔ `INSERT` 变动流水。
        4. 提交 Spring `@Transactional` 事务。
        5. **释放 Redis 分布式锁**。（_注：严格保证锁包住事务，彻底杜绝事务未提交锁已释放导致的并发脏读与数据覆盖_）。

### 2.2 基于 Kafka 的事件驱动解耦机制 (EDA)
+ **业务场景**：粮情录入后需立即报警、通风结束后需计算复杂能耗报表。此类重度计算不能阻塞前端用户的操作。
+ **技术方案**：
    - **Topic 1**: `topic_grain_condition_recorded` (粮情已录入事件)。报警微服务订阅此Topic，比对阈值，触发 WebSocket 向库主任推送报警。
    - **Topic 2**: `topic_ventilation_completed` (通风已完成事件)。报表微服务订阅此Topic，静默抓取起始/结束电表读数，计算“吨粮降温能耗”并落入数仓体系。

## 三、 详细功能需求说明 (按领域划分)
### 📌 领域一：基础主数据领域 (Master Data)
**1.1 仓储空间建模**

+ **功能描述**：定义粮食存放的物理与虚拟空间。
+ **字段要求**：
    - **库区管理 (Warehouse)**：粮库名称、地址、总设计容量。
    - **仓房管理 (Bin)**：归属库区、仓房编号（如01仓）、仓型（高大平房仓/浅圆仓）、仓容。
    - **货位管理 (Position)**：归属仓房、货位编号（如A区/B区）、货位容量。**（所有实际库存绑定至货位级别）**。

### 📌 领域二：极简出入库领域 (Inbound/Outbound)
**2.1 手工出入库单**

+ **操作角色**：保管员 / 登记员
+ **功能描述**：录入最核心的库存变动指令。
+ **界面表单**：
    - 业务类型：单选（入库 / 出库）。
    - 存储位置：**级联选择**【库区 ➔ 仓房 ➔ 货位】。
    - 粮食信息：粮食品种、收获年份、等级。
    - 变动重量：输入具体吨数（精确到小数点后三位）。
+ **交互逻辑**：点击【提交校验】，触发 `2.1章节` 的底层分段锁扣减机制。若当前“货位”正有其他单据处理中，前端 Loading 等待或提示“当前货位业务繁忙，请重试”。

### 📌 领域三：粮情检测领域 (Grain Condition)
**3.1 日常查仓记录录入**

+ **操作角色**：保管员
+ **功能描述**：支持保管员根据线下查仓结果，建立各仓房的粮情健康档案。
+ **界面表单（按天/仓房维度）**：
    - **环境温湿度**：仓外温(℃)、仓外湿(%)、仓内温(℃)、仓内湿(%)。
    - **粮温信息**：最高粮温(℃)、最低粮温(℃)、平均粮温(℃)。
    - **虫情气体**：虫粮等级（下拉）、虫害描述（文本）、O2浓度(%)、CO2浓度(%)。
    - **现场附件**：异常备注说明、支持多图上传（现场水浸/结露照片）。
+ **交互逻辑**：录入保存后，系统流转至正常状态。后台静默发送 Kafka 消息。

**3.2 粮情异常实时报警 (基于WebSocket)**

+ **操作角色**：库主任 / 安全员
+ **功能描述**：无需刷新页面，接收系统自动判定的异常报警。
+ **判定规则字典**：系统设置全局阈值（如：任一仓房最高粮温 > 28℃，或 CO2 > 警戒值）。
+ **展现形式**：触发阈值时，PC端右下角弹出红色报警卡片，伴随提示音，点击可查看具体的仓房与粮情明细。

### 📌 领域四：通风作业领域 (Ventilation)
**4.1 通风任务状态机管控**

+ **操作角色**：保管员
+ **状态流转**：`待执行` ➔ `作业中` ➔ `已完成`
+ **核心动作**：
    - **开始作业**：录入【开始时间】、【起始电表读数】、【初始平均粮温/水分】。
    - **结束作业**：录入【结束时间】、【结束电表读数】、【结束平均粮温/水分】。
+ **交互逻辑**：点击结束作业后，主表状态更新为完成。后台静默发送 Kafka 消息，触发统计报表异步计算。

### 📌 领域五：数据统计与报表领域 (Analytics)
**5.1 实时库存动态台账**

+ **数据来源**：基于实时 MySQL 聚合（得益于强一致性的库存变动日志）。
+ **展现形式**：
    - **汇总层**：按【库区】和【仓房】展示总储量、空余库容比例（进度条）。
    - **下钻层**：点击某个仓房，展开展示内部各【货位】的当前存粮品种与精准吨数。

**5.2 ****📈**** 粮情历史趋势视图 (核心业务亮点)**

+ **操作角色**：库主任 / 保管员
+ **功能描述**：可视化辅助判断发热霉变风险。
+ **展现形式**：
    - **组合筛选器**：选择【仓房】 + 【时间区间（如最近30天）】。
    - **三温比对折线图**：X轴为日期，Y轴为温度。将该仓房历次录入的**“仓外温”、“仓内温”、“平均粮温”**三条曲线绘制在同一张图内。
    - _业务价值_：帮助管理者直观发现“气温下降但粮温异常突增”的“剪刀差”现象。
    - **明细数据表**：折线图下方附带该时段的详细列表（含虫情、气体信息），支持 Excel 一键导出。

**5.3 通风效能分析报表**

+ **数据来源**：由 Kafka 消费者计算完毕的数仓结果表。
+ **展现形式**：列表展示历史通风记录。包含核心计算指标：总耗电量、降温幅度(℃)、降水幅度(%)、**吨粮降温能耗(度/吨·℃)**，用于评价各仓房保管员的通风技术水平。

## 四、 非功能性需求 (NFR)
1. **高并发性能**：基于 `Warehouse_Bin_Position` 分段锁设计，出入库单笔操作响应时间必须在 500ms 内，单仓房内不同货位支持完全无阻塞并发读写。
2. **安全性与幂等性**：出入库前端提交按钮需做防抖（Debounce）；后端 API 需实现全局幂等性校验，防止网络卡顿导致的重复扣减。
3. **IoT 扩展预备（架构要求）**：粮情模块的数据库表结构必须包含 `data_source` 字段（枚举值：`manual` 手工 / `iot` 传感器），以保障二期接入自动化测温电缆时，仅需新增 Kafka 生产者网关，本期开发的分析与图表功能**无需重构即可完全复用**。
